13 Ruuvi Firmware version 3. Built on top of Nordic SDK 15, uses both Ruuvi and external repositories as submodules.
14 Under development, please follow [Ruuvi Blog](https://blog.ruuvi.com) for details. The project is in alpha stage,
15 build maybe broken in obvious ways and not build at all or in non-obvious ways and fail on some Ruuvi boards.

17 # Setting up
18 ## SDK 15.3
19 Download [Nordic SDK15.3](https://developer.nordicsemi.com/nRF5_SDK/nRF5_SDK_v15.x.x/) and install it at the project root.
20 If you're working on multiple nRF projects, you can (and should) use softlinking instead.
21 
22 ## Submodules
23 Run `git submodule update --init --recursive`. This will search for and install the other git repositories referenced by this project. If any of the submodules has a changed remote, you'll need to run `git submodule sync --recursive` and again `git submodule update --init --recursive` to update the modules from new remotes.
24 
25 ## Toolchain
26 ARMGCC is used for [Jenkins builds](http://jenkins.ruuvi.com/job/ruuvi.firmware.c/), it's recommended to use SES for developing. You can make the project and all variants by runnning "make" at top level of this repository.
27 
28 Segger Embedded Studio can be set up by installing [nRF Connect for Desktop](https://www.nordicsemi.com/?sc_itemid=%7BB935528E-8BFA-42D9-8BB5-83E2A5E1FF5C%7D)
29 and following Getting Started plugin instructions.
30 
31 Start SES and open `ruuvi_ruuvi.firmware.c.emProject` at root level, each of the target boards is in their own project.
32 
33 ## Code style
34 Code is formatted with [Artistic Style](http://astyle.sourceforge.net).
35 Run `astyle --project=.astylerc ./target_file`. To format the entire project,
36 ```
37 astyle --project=.astylerc "src/main.c"
38 astyle --project=.astylerc --recursive "src/application_config/*.h"
39 astyle --project=.astylerc --recursive "src/tasks/*.c"
40 astyle --project=.astylerc --recursive "src/tasks/*.h"
41 astyle --project=.astylerc --recursive "src/tests/*.c"
42 astyle --project=.astylerc --recursive "src/tests/*.h"
43 ```

49 ### PVS
50 Obtain license and software from [Viva64](https://www.viva64.com/en/pvs-studio/).
51 
52 Make runs PVS Studio scan and outputs results under doxygen/html/fullhtml.
53 
54 This results into hundreds of warnings, it is up to you to filter the data you're interested in. For example you probably want to filter out warnings related to 64-bit systems.
55 
56 ### Sonar scan
57 Travis pushes the results to [SonarCloud.IO](https://sonarcloud.io/dashboard?id=ruuvi_ruuvi.firmware.c).
58 SonarCloud uses access token which is private to Ruuvi, you'll need to fork the project and setup
59 the SonarCloud under your own account if you wish to run Sonar Scan on your own code.

61 # Running unit tests
62 ## Ceedling
63 Unit tests are implemented with Ceedling. Run the tests with
64 `ceedling test:all`
65 
66 ### Gcov
67 Ceedling can also generate Gcov reports with `ceedling gcov:all utils:gcov`.
68 The report can be found under _build/artifacts/gcov_.
69 
70 ## Unit test continuous integration
71 Travis will fail the build if unit test fails and Gcov results will get pushed to SonarCloud.

73 # Usage
74 Compile and flash the project to your board using SES. Instructions on how to use a bootloader will be added later on.
75 The project is not yet in a useful state for any practical purpose other than learning.
76 
77 Note: You should erase your board entirely in case there is a bootloader from a previous firmware.

79 # How to contribute
80 Please let us know your thoughts on the direction and structure of the project. Does the project help you to understand how to build code on RuuviTag?
81 Is the structure of the project sensible to you?
82 
83 If you want to assist in the project maintenance by fixing some issues _doxygen.error_ is
84 a good place to look for code which needs better commenting. Project badges at the top of the
85 readme point to issues which range from trivial clarifications to complex refactoring.
86 
87 If you want to add new features, please discuss the feature first, and then create ceedling
88 unit tests for the functionality. Once the functionality is agreed and testable in can be integrated
89 into project.
